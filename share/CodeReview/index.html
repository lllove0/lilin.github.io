什么是 Code Review

                           为什么叫"礼仪"

为什么需要 Code Review

开发人员的"盲点"

  1. 程序是照你写的跑，不是照你想的跑
  2. 程序照你想的跑，但你想的是错的

程序是属于团队的

Code Review 的目标

保证代码质量

统一代码风格

增强代码阅读性

指引新同学

普及知识

验证业务

如何做 Code Review

  1. Review 前的工作准备
  2. Review 进行时
  3. Review 的收尾

Review 前的准备工作

  1. Reivew 通告

2、代码自检

JShint

JavaScript 规范
(argo\doc\argo-in-depth\guideline\JavaScript-standard.md)

AngularJs 最佳实践
(https://github.com/johnpapa/angular-styleguide/tree/master/a1)

3、 设施基础检查

产品是否可展示

IDE 的字体大小

Review 进行时

  1. 开始前再次通告
  2. 普及业务背景
  3. 描述功能，代码设计
  4. 根据代码框架层层描述
  5. 如实记录 Review Comments
  6. 遇到争议较大的问题，应先记录再讨论

Review 过程中不得修改代码

善用 FIXME 或 TODO

FIXME : 有 bug 需要修改

TODO : 还未完善的功能

参加 Review 的人要做什么

大胆提问

程序是否满足业务需求

程序是否满足未来的扩充性

代码是否有坏味道

代码易读性

Error Handing

Review 收尾

  1. 当天出 Review 摘要
  2. 归纳总结 Review Comments
  3. @参加人员 确认
  4. 被@的人需要确认回复
  5. 如实记录 Review Comments
  6. 如果修改的内容比较多，需安排多次 Review

NG

  1. 基础问题太多
  2. 设计方面有重大的问题
  3. 业务上有较大的出入
  1. NG Review 摘要
  2. 重新安排 Review

总结

  1. 审核清单
  2. 提前准备不能省，让会议效率最优化
  3. 频繁且小范围的进行 Review
  4. 快速提升团队的整理水平
  5. 通过 Review 摘要来追踪会议记录与代码事项
  6. 建立团队文化和开发规范
  7. Review 的心态要正确
